# Glossary

Definitions of domain-specific terms used throughout this project.
Only the most important, frequently used terms and concepts should be placed here.
It's purpose is to provide a quick context for the agents reading the docs about terms which are not immediately obvious.
Keep it around 15-20 terms max, don't make this an encylopedia, the rest of the docs are discoverable through normal search.

---

## How to Use

- Link to specific terms using the format: `[Term](./glossary.md#term-name)`
- Terms are alphabetized
- Each term has an anchor matching its kebab-case name

## Project Terms

### Agent

An AI session identity (e.g., "Adele", "Brian"). Each agent has a workspace, can hold one role at a time, and is assigned to a human. Agents are claimed per terminal session and released when done.

**See also:** [Agent Lifecycle](./understand/agent-lifecycle.md)

### Audit Trail

JSON records of every agent action — file reads, writes, blocks, role changes — stored in `dydo/_system/audit/`. Used for replay, debugging, and accountability.

**See also:** [Audit System](./reference/audit-system.md)

### Claim

The act of an agent binding to a terminal session via `dydo agent claim`. Only one agent per session; only one session per agent.

### Dispatch

Handing work to another agent via `dydo dispatch`. Creates an inbox item and optionally launches a new terminal. Can be `--wait` (blocking) or `--no-wait` (fire-and-forget).

**See also:** [Dispatch and Messaging](./understand/dispatch-and-messaging.md)

### Guard

The `dydo guard` hook that intercepts every tool call. Enforces staged access, role-based permissions, off-limits patterns, and bash safety checks. Exit code 0 = allowed, 2 = blocked.

**See also:** [Guard System](./understand/guard-system.md)

### Hub File

An `_index.md` file in each documentation folder. Serves as the table of contents for that folder. Auto-generated by `dydo fix`.

### Inbox

A per-agent queue of dispatched work items. Agents check their inbox with `dydo inbox show` and clear processed items with `dydo inbox clear`.

### JITI

Just-In-Time Information. The documentation philosophy: agents don't read everything upfront — they navigate to what they need for the current task.

### Mode File

A role-specific instruction file at `dydo/agents/{name}/modes/{role}.md`. Contains must-reads, work guidance, and completion instructions for that role.

### Must-Read

A document with `must-read: true` in its frontmatter. The guard blocks writes until the agent has read all must-read files linked from their mode file.

### Off-Limits

File patterns in `dydo/files-off-limits.md` that block ALL agent access — secrets, credentials, system state files. A whitelist section can carve exceptions.

### Role

A permission set (e.g., `code-writer`, `reviewer`) that determines what files an agent can write. Defined in `.role.json` files. Set via `dydo agent role`.

**See also:** [Roles and Permissions](./understand/roles-and-permissions.md)

### Snapshot

A project state capture (files, folders, doc links) stored in each audit session. Used for replay and compacted via baseline+delta compression.

### Staged Access

The guard's three-stage onboarding: Stage 0 (no identity — bootstrap only), Stage 1 (claimed, no role — mode files only), Stage 2 (claimed + role — full access per role permissions).

### Task

A tracked unit of work with a lifecycle: pending → in-progress → review-pending → approved/rejected. Stored as markdown in `dydo/project/tasks/`.

### Template Addition

A markdown file in `dydo/_system/template-additions/` injected into agent templates via `{{include:name}}` tags. Allows extending agent workflows without editing templates directly.
