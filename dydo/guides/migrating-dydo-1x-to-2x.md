# Migrating a Project from dydo 1.x to 2.x

A self-contained, ordered procedure for an AI coding agent to migrate an existing project's
dydo workspace from the 1.x generation to the current 2.x generation. Execute it **in the target
project** (the repo being migrated), not in the dydo source repo.

> **Read this first.** This guide was written from the dydo **2.x source repo** as the ground
> truth for current conventions. The author did not have the target project's 1.x tree in front of
> them. Every place that depends on a 1.x-specific detail is marked **[VERIFY]** — stop and check
> your actual tree there before acting. When in doubt, prefer letting the **dydo binary** do the
> work (`dydo template update`, `dydo roles reset`, `dydo sync`, `dydo fix`) over hand-editing, and
> confirm each step with `dydo check`.

---

## 0. What changed 1.x → 2.x (summary)

Three classes of change, plus tooling:

**A. Folders — `dydo/project/` gained record types, `_system/` was formalized.**
- The Notion two-way sync work (Decision 025) introduced a full PM "spine": `project/campaigns/`,
  `project/sprints/`, `project/sprint-tasks/`, `project/releases/`, and `project/future-features/`
  as first-class record folders, plus a date-nested `project/changelog/YYYY/YYYY-MM-DD/` archive.
- `dydo/_system/` is the 2.x home for framework machinery: **`_system/roles/*.role.json`**
  (data-driven role definitions — Decision 024), `_system/templates/` (mode + workflow templates),
  `_system/sync-model.json`, `_system/types.json`, `_system/notion_sync/`, and the gitignored
  `_system/.local/` (queues, worktrees, sync state, watchdog anchors).
- `dydo/reference/roles/*.md` are **generated** human-readable role docs.

**B. Frontmatter — records gained identity + provenance fields.**
- **Issues** now carry `title`, `id`, `type: issue`, `severity`, `found-by`, and a **provenance
  triad** `found-by-agent` / `found-by-vendor` / `found-by-model`, plus `date` (and `resolved-date`
  once resolved).
- **Tasks** now carry a provenance pair `assigned-vendor` / `assigned-model` alongside `assigned`.
- Sprint / sprint-task / campaign / release records carry the spine fields the Notion board needs
  (`status`, `priority`, `work-type`, `sprint`, `seq`, `start`/`end`, `blocked-by`, `needs-human`,
  etc.).

**C. Roles are data, not hardcoded matrices.** As of 2.0 (Decision 024) dydo no longer enforces a
per-role writable/read-only **path matrix** at runtime. The guard enforces **universal off-limits +
nudges** for every agent; a role's read-only-ness is expressed by its compiled **native tool
allowlist** (`dydo sync` emits reviewer/inquisitor/sprint-auditor with no Edit/Write tool). Role
definitions live as `.role.json` and are **compiled** into native Claude/Codex agents by `dydo sync`.

**D. Tooling a 1.x user won't know:** `dydo sync`, `dydo roles reset`, `dydo model cap`/`uncap`,
`dydo notion connect`/`sync`/`reset`, `dydo read`, `dydo hand raise`/`lower`, plus a couple of
`gap_check.py` updates (Section 6).

## 1. Prerequisites and safety net

1. **Branch first.** `git checkout -b chore/dydo-2x-migration`. Do all work on a branch; the
   migration touches many files.
2. **Commit or stash any dirty state** so the diff you produce is only the migration.
3. **Upgrade the dydo binary to 2.x** (this is the engine that regenerates everything):
   ```bash
   npm install -g dydo            # if installed via npm
   # or
   dotnet tool update -g dydo     # if installed via dotnet tool
   dydo version                   # confirm it reports a 2.x version
   ```
4. **Baseline the current state** so you can measure progress:
   ```bash
   dydo check > /tmp/dydo-check-before.txt 2>&1 || true
   git add -A && git commit -m "chore: pre-migration baseline" --allow-empty
   ```
   Expect `dydo check` to report violations right now — that is the migration backlog.

---

## 2. Let the binary rebuild framework-owned files (do this before hand edits)

Most of the 2.x scaffolding is framework-owned and regenerated by the CLI. Run these **in order**;
they are idempotent.

1. **Update templates + framework docs to 2.x:**
   ```bash
   dydo template update --diff     # preview
   dydo template update            # apply; re-anchors your {{include:...}} tags
   ```
   This refreshes `dydo/_system/templates/`, the framework guides/reference, `types.json`, and the
   `sync-model` template. User-added `{{include:...}}` tags are preserved; **non-include hand edits
   to framework-owned files are lost** — if you customized a framework doc's prose, save it first.
   **[VERIFY]** review `--diff` output before applying.

2. **Regenerate role definitions with the NEW binary:**
   ```bash
   dydo roles reset          # regenerate base .role.json files in dydo/_system/roles/
   # dydo roles reset --all  # ONLY if you have no custom roles worth keeping (wipes custom too)
   ```
   Use plain `dydo roles reset` if you have custom roles; it rewrites only the base ones.

3. **Compile roles into native agents:**
   ```bash
   dydo sync                 # emits .claude/agents/*, .claude/skills/*, and (if codex) .codex/*
   ```

4. **Rebuild auto-generated hub/index files:**
   ```bash
   dydo fix                  # kebab-case renames, wikilink conversion, (re)creates hub files
   dydo index                # regenerate dydo/index.md + folder _index.md hubs
   ```

**Verify Section 2:** `dydo/_system/roles/` contains one `.role.json` per role; `dydo/reference/roles/`
has a generated `*.md` per role; `.claude/agents/` (and `.codex/agents/` if codex) were written;
`git status` shows the regenerated files.

## 3. Folder layout — the 2.x target and how to reconcile

This is the canonical 2.x `dydo/` layout. Compare it against the target tree and **move any
misplaced PM objects into these folders.** The spine loader pools `*.md` recursively per record
folder and keys each row by **filename stem**, so moves are safe **as long as you never create two
files with the same stem under one record type** (a duplicate stem crashes the sync).

```
dydo/
├── index.md                      # generated hub (do NOT hand-edit)
├── _system/                      # framework machinery (mostly agent-off-limits)
│   ├── roles/*.role.json         # data-driven role defs (dydo roles reset)
│   ├── templates/                # mode-*.template.md, agent-workflow.template.md
│   ├── sync-model.json           # Notion spine model (generated; create-only)
│   ├── types.json                # frontmatter `type:` vocabulary
│   ├── notion_sync/              # canonical sync mirror state
│   └── .local/                   # gitignored: queues, worktrees, sync, watchdog anchors
├── agents/<Name>/                # per-agent workspace, inbox/, archive/, modes/
├── guides/                       # browsable how-to docs  → Notion docs mirror
├── reference/                    # browsable reference     → Notion docs mirror
│   └── roles/*.md                # generated role docs
├── understand/                   # architecture / concept prose → docs mirror
└── project/                      # PM RECORDS → Notion spine databases (excluded from docs mirror)
    ├── issues/                   # Issue rows; resolved/ subfolder partitions status:resolved
    │   └── resolved/
    ├── tasks/                    # Task rows (active)
    ├── decisions/                # Decision records (NNN-title.md)
    ├── sprints/                  # Sprint rows                      [2.x — Decision 025]
    ├── sprint-tasks/             # SprintTask rows                  [2.x — Decision 025]
    ├── campaigns/                # Campaign rows                    [2.x — Decision 025]
    ├── releases/                 # Release rows                     [2.x — Decision 025]
    ├── future-features/          # FutureFeature / idea records     [2.x]
    ├── backlog/                  # backlog items (see DR 034 note below)
    ├── pitfalls/                 # Pitfall records
    └── changelog/YYYY/YYYY-MM-DD/  # done-task archive, date-nested [2.x]
```

**Migration actions:**

1. **Create any missing record folders** listed above that your project will use. Empty ones are
   fine; `dydo fix` will scaffold their `_index.md` / folder-meta hub. Do not create status
   subfolders you don't need (e.g. only `issues/resolved/` exists by convention; open/triage issues
   live at the `issues/` root).

2. **[VERIFY] Relocate PM objects that a 1.x tree kept elsewhere.** Likely 1.x deltas to look for
   (confirm against your actual tree):
   - Sprint / campaign / release notes kept as loose docs or under a different folder → move into
     `project/sprints/`, `project/campaigns/`, `project/releases/`.
   - A flat `changelog/` (not date-nested) → move entries under `changelog/YYYY/YYYY-MM-DD/`.
   - Role config kept outside `_system/roles/` → superseded by `dydo roles reset` (Section 2).
   Move with `git mv` to preserve history. After **every** batch of moves, run `dydo fix` so hub
   indexes and wikilinks rebuild, then `dydo check`.

3. **DR 034 (PM Record Taxonomy) — apply only if your dydo binary already ships it. [VERIFY]**
   Decision 034 folds `backlog/` into `tasks/backlog/` (a `Task` at `status: backlog`), promotes
   `future-features/` to a `FutureFeature` record type, and yanks `inquisitions/`. As of this
   writing that reshape is **partially landed** in dydo mainline (the `sync-model` gained the `Task`
   and `FutureFeature` types; the file moves had not shipped). **Do not pre-emptively move
   `backlog/` or delete `inquisitions/`** unless your `dydo` version's own `dydo init` scaffolds
   that shape — check by running `dydo init` in a scratch dir and inspecting its `project/` tree.
   If your binary does not yet ship the moves, keep `backlog/`, `future-features/`, and
   `inquisitions/` as-is; they remain valid.

**Verify Section 3:** `dydo check` no longer reports "missing hub" or "orphan folder" errors for
the record folders; `git status` shows moved files under their 2.x paths; no duplicate filename
stems exist within any one record type (`find dydo/project/<type> -name '*.md' | sed 's#.*/##' |
sort | uniq -d` returns nothing).

## 4. Frontmatter — bring each record type up to the 2.x shape

For each record type below: **required** fields must be present; **provenance** fields are 2.x
additions. The safest way to learn the exact current shape your binary expects is to create a
throwaway record and copy its frontmatter:

```bash
dydo issue create --title "shape probe" --area general --severity low --summary "probe"
dydo task create shape-probe --area general --description "probe"
# inspect the generated files, copy the frontmatter shape, then delete the probes
```

### 4.1 Issue (`project/issues/*.md`)

Current 2.x shape (from `IssueCreateHandler`):

```yaml
---
title: <one-line title>
id: <integer, unique, monotonic>
area: <backend|frontend|microservices|platform|general|understand|guides|reference|project>
type: issue
severity: <low|medium|high|critical>
status: <open|resolved>
found-by: <manual|inquisition|review>
found-by-agent: <AgentName>        # provenance triad — 2.x; present only if created by an agent
found-by-vendor: <claude|codex|...>
found-by-model: <model display name|unknown>
date: <YYYY-MM-DD>
resolved-date: <YYYY-MM-DD>        # only on resolved issues
---
```

Migration for existing 1.x issues (**[VERIFY]** which of these your 1.x issues already had):
- Add `type: issue` if absent.
- Add `title:` (copy the H1) and `id:` (a unique integer; keep existing numeric prefixes in the
  filename as the id). Ensure ids are unique across `issues/` **and** `issues/resolved/`.
- Add `severity:` (default `medium` if unknown) and `found-by:` (default `manual`).
- Add the **provenance triad** for issues created going forward. For historical issues you cannot
  attribute, either omit the triad (the create handler omits it when there is no agent) or set
  `found-by-agent: unknown` / `found-by-vendor: unknown` / `found-by-model: unknown`. **[VERIFY]**
  whether your `dydo check` warns on a missing triad; if it does not warn, leave historical issues
  without it rather than inventing provenance.
- Resolved issues live under `issues/resolved/` with `status: resolved` and `resolved-date:`.

### 4.2 Task (`project/tasks/*.md`)

Current 2.x shape (from `TaskCreateHandler`):

```yaml
---
area: <area>
name: <task-name matching filename stem>
status: <pending|review-pending|...>
created: <ISO-8601 timestamp>
assigned: <AgentName|unassigned>
assigned-vendor: <claude|codex|...>   # 2.x; present only if created by an agent
assigned-model: <model|unknown>       # 2.x
updated: <ISO-8601 timestamp>          # maintained by the lifecycle
---
```

Migration: ensure `name:` matches the filename stem; add `assigned-vendor` / `assigned-model` for
agent-created tasks (omit or `unknown` for historical). Do **not** add a `type: task` field unless
your binary's `dydo task create` emits one (it currently does not).

### 4.3 Decision (`project/decisions/NNN-title.md`)

```yaml
---
area: <area>
type: decision
status: <accepted|proposed|superseded|...>
date: <YYYY-MM-DD>
participants: [name, name]
---
```

Migration: add `type: decision` and `status:` if 1.x decisions lacked them.

### 4.4 Sprint (`project/sprints/*.md`)

```yaml
---
title: <Sprint title>
campaign: <campaign-name or empty>
seq: <integer>
start: <YYYY-MM-DD>
end: <YYYY-MM-DD>
status: <planned|active|done>
gate-result: <free text — review/audit verdict>
area: project
type: context
---
```

### 4.5 Sprint-task (`project/sprint-tasks/*.md`)

```yaml
---
title: <SprintTask title>
sprint: <parent sprint stem>
status: <todo|in-progress|in-review|done>
priority: <Low|Medium|High|Urgent>
work-type: <feature|fix|...>
blocked-by:            # stem(s) this is blocked by, or empty
due:                   # optional
needs-human: <true|false>
area: <area>
type: context
---
```

### 4.6 Backlog / future-feature (current shipped shape)

Backlog item:
```yaml
---
area: <area>
type: context
name: <item-name>
status: <open|...>
---
```
Future-feature:
```yaml
---
area: <area>
type: concept
status: <idea|...>
---
```
(If your binary ships DR 034, these become `Task`/`FutureFeature` records — see Section 3.3.)

**Verify Section 4:** `dydo check` reports no frontmatter errors (missing required field, bad
`type`, bad `area`, bad `severity`). `area` must be one of: `frontend, backend, microservices,
platform, general, understand, guides, reference, project`. Every `type` value must exist in
`dydo/_system/types.json`.

## 5. Roles and native-agent regeneration

1. Confirm `dydo/_system/roles/` holds `.role.json` for each role you use. `dydo roles list` lists
   loaded roles.
2. If you carried custom roles from 1.x, re-check their `.role.json` against the 2.x schema
   (`dydo/guides/customizing-roles.md`). Note the semantics change: `writablePaths` / `readOnlyPaths`
   **no longer enforce runtime write-RBAC** — they only inform how `dydo sync` compiles the native
   tool allowlist (a role writable only to `{self}` compiles read-only; broader compiles read-write).
3. Re-run `dydo sync` after any role edit so the native `.claude/agents/*` and `.codex/agents/*`
   are regenerated. **Stale 1.x agent files are the #1 cause of post-migration weirdness** — always
   regenerate with the new binary rather than keeping the old compiled artifacts.

**Verify Section 5:** `dydo sync` exits 0; read-only roles (reviewer, inquisitor, sprint-auditor)
have no Edit/Write in their compiled agent definitions; `dydo validate` passes.

## 6. Update the project's copy of `gap_check.py`

`DynaDocs.Tests/coverage/gap_check.py` (path may differ — **[VERIFY]** the target's coverage dir) is
a **copy** in each project, so upstream fixes must be hand-applied. Apply these two, in either order.
Both are small and localized.

### 6.1 Staleness scan must cover `Sync/`, `Templates/`, and top-level `Program.cs` (issue #0217)

Before this fix, the staleness check (which decides whether to skip re-running tests) scanned only a
hardcoded set of source dirs with a single `*.cs` glob — so edits to `Sync/`, `Program.cs`, or the
embedded-resource files under `Templates/` re-scored **stale coverage as fresh** (false green).

Edit the module-level constants block (near the top, after `GENERATED_PATTERNS`). Change:

```python
# Source directories to check for staleness (relative to ROOT)
SOURCE_DIRS = ["Commands", "Services", "Models", "Rules", "Utils", "Serialization"]
SOURCE_GLOBS = ["*.cs"]
```
to:
```python
# Source directories to check for staleness (relative to ROOT)
SOURCE_DIRS = ["Commands", "Services", "Models", "Rules", "Utils", "Serialization", "Sync", "Templates"]
SOURCE_FILES = ["Program.cs"]
SOURCE_GLOBS = ["*.cs"]
SOURCE_DIR_GLOBS = {"Templates": ["*"]}
```

Then in `_find_changed_files_since(threshold_mtime)`, change the per-dir loop so it uses the per-dir
glob map and filters out directories, and add a loop over `SOURCE_FILES`. Replace:

```python
        for glob in SOURCE_GLOBS:
            for src in d.rglob(glob):
                if is_generated(str(src)):
                    continue
                if src.stat().st_mtime > threshold_mtime:
                    changed.append(str(src.relative_to(ROOT)))
```
with:
```python
        for glob in SOURCE_DIR_GLOBS.get(src_dir, SOURCE_GLOBS):
            for src in d.rglob(glob):
                if not src.is_file():
                    continue
                if is_generated(str(src)):
                    continue
                if src.stat().st_mtime > threshold_mtime:
                    changed.append(str(src.relative_to(ROOT)))

    for src_file in SOURCE_FILES:
        src = ROOT / src_file
        if src.exists() and src.stat().st_mtime > threshold_mtime:
            changed.append(str(src.relative_to(ROOT)))
```
(The new `for src_file in SOURCE_FILES:` block sits between the `SOURCE_DIRS` loop and the existing
`test_dir = ROOT / "DynaDocs.Tests"` block.)

Notes for adapting to the target: `Templates` uses `["*"]` because its files are embedded resources,
not `.cs` (a `*.cs` glob matches nothing there); the `is_file()` guard is required because `rglob("*")`
also yields directories. **[VERIFY]** the target's real source-dir names and adjust `SOURCE_DIRS` /
`SOURCE_FILES` / `SOURCE_DIR_GLOBS` to match that project's tree (e.g. a different top-level entrypoint
than `Program.cs`, or different non-`.cs` resource dirs).

### 6.2 Propagate the test exit code through the gate (issue #0169)

Before this fix the gate exited 0 when `dotnet test` failed as long as the tier check passed —
silently green-lighting flakes and regressions. In `main()`:

1. Initialize `tests_ok` up front (the staleness-skip path never assigns it, so without this the new
   reference raises `UnboundLocalError`). Right after `args = parser.parse_args()` / before the
   "Decide whether to run tests" block, add:
   ```python
   # tests_ok stays True on the staleness-skip path (no test process ran, so
   # there is no failure to propagate). The two run_tests() branches below
   # overwrite it with the actual exit-code-derived value.
   tests_ok = True
   ```
2. Fold `tests_ok` into the exit decision. Replace the final:
   ```python
   if has_failures or registry_errors:
       sys.exit(1)
   sys.exit(0)
   ```
   with:
   ```python
   if not tests_ok:
       tier_check = "fail" if has_failures else "pass"
       print(f"\n  [RESULT] Tests failed (see exit-code line above). "
             f"Tier check: {tier_check}. Gate FAILS.")
   if has_failures or registry_errors or not tests_ok:
       sys.exit(1)
   sys.exit(0)
   ```

**Verify Section 6:** `python <coverage-dir>/gap_check.py --force-run` runs to completion and exits
0 on a green tree; touching a file under `Sync/` or `Templates/` (or `Program.cs`) and re-running
**without** `--force-run` now reports files changed and re-runs tests instead of skipping.

## 7. Gotchas / order matters

- **Regenerate, don't hand-port, compiled artifacts.** Always run `dydo roles reset` → `dydo sync`
  with the **new** binary. Old `.claude/agents/*` / `.codex/*` from 1.x will silently misbehave.
- **Run `dydo fix` (and `dydo index`) *after* moving PM objects**, not before — the hub/index files
  are derived from the tree, so move first, then rebuild.
- **Never hand-edit auto-generated `_index.md` / `index.md` / folder-meta hubs.** They are
  regenerated by `dydo fix` / `dydo index`; hand edits are overwritten and will fight the tool.
- **Filename-stem uniqueness per record type.** The spine loader keys rows by stem across all
  subfolders of a type. Two files with the same stem (e.g. the same task archived on two dates)
  crash the sync — disambiguate before moving.
- **`dydo template update` loses non-include hand edits** to framework-owned files. Preview with
  `--diff`; re-add customizations as `{{include:...}}` blocks so they survive future updates.
- **`sync-model.json` is create-only / agent-off-limits.** `dydo template update` refreshes the
  *template*, but the generated `dydo/_system/sync-model.json` is seeded only when missing. If you
  need the new object types live, that is a deliberate re-provision (`dydo notion sync` /
  `dydo notion reset`), not an automatic effect of the migration. Do not hand-edit it.
- **Do the DR 034 taxonomy reshape only if your binary ships it** (Section 3.3). Guessing ahead of
  the binary creates a tree the installed `dydo` doesn't understand.
- **Notion is optional.** If the target project never used Notion sync, you can skip all
  `dydo notion *` steps; the repo files remain the single source of truth.

## 8. Verification checklist (run at the end)

Run every item; all must pass before declaring the migration complete.

- [ ] `dydo version` reports a 2.x version.
- [ ] `dydo check` exits 0 (no frontmatter, link, naming, hub, or orphan errors). Diff against
      `/tmp/dydo-check-before.txt` to confirm every baseline violation is gone.
- [ ] `dydo validate` passes (dydo.json, role files, agent assignments all valid).
- [ ] `dydo roles list` shows every expected role; `dydo/_system/roles/*.role.json` present.
- [ ] `dydo sync` exits 0 and (re)wrote `.claude/agents/*` (+ `.codex/*` if codex is enabled);
      read-only roles have no Edit/Write tool.
- [ ] `dydo fix` and `dydo index` run clean (a second run produces **no** diff — proves hubs are
      stable/idempotent).
- [ ] Every PM record sits in its 2.x folder (Section 3); no duplicate filename stems within a
      record type.
- [ ] Issues carry `title` / `id` / `type: issue` / `severity` / `found-by` (+ provenance triad on
      agent-created ones); tasks carry `assigned-vendor` / `assigned-model` on agent-created ones.
- [ ] `gap_check.py` has both updates from Section 6; `python <coverage-dir>/gap_check.py
      --force-run` exits 0 on a green tree, and a touch under `Sync/`/`Templates/`/`Program.cs`
      re-triggers tests without `--force-run`.
- [ ] Full test suite green (`dotnet test` or the project's runner).
- [ ] `git status` / `git diff` reviewed — the branch contains only intended migration changes.
- [ ] (If Notion in use) `dydo notion sync --dry-run` produces a sane plan with no schema-drift
      surprises.
