# dydo 2.0 — Hardening, Cleanup & Onboarding Backlog

Deferred work surfaced while landing the 2.0 campaign + Notion sync. Not blocking; to
be picked up during the refine phase. Grouped by theme.

## Install & legacy

- **Update the installed global `dydo` to the 2.0 build.** The `~/.dotnet/tools/dydo.exe`
  on PATH is still pre-2.0: it runs for the guard hook and `dydo agent claim/dispatch`,
  and it is what still **writes audit files** (the 2.0 code produces none) and re-seeds the
  dropped `mode-inquisitor/judge` templates. Root fix for most of the "legacy" feel.
- **Purge audit cruft.** ~1644 audit `.events`/`.json` files are tracked in git under
  `dydo/_system/audit/`. `git rm` them, add `dydo/_system/audit/` to `.gitignore`, and remove
  any orphaned audit test fixtures (`DynaDocs.Tests/Fixtures/audit-*`). Do AFTER the install so
  nothing re-writes them.
- **Legacy-sweep inquisition (run once the 2.0 tool is installed).** A dedicated inquisition
  for dead/legacy code, files, and docs across the repo — audit remnants, dropped-role
  reference docs (`dydo/reference/roles/inquisitor.md`, `judge.md`), stale comments, orphaned
  config keys, etc.

## Notion — open-source onboarding

- **`dydo notion connect` (or `setup`) command + onboarding docs.** dydo is open-source; other
  users must set up their own Notion internal integration + token. Provide a guided command
  (paste your integration secret, pick/confirm the parent page, store it) and a tutorial doc
  covering: create an internal integration, share the parent page with it, set `DYDO_NOTION_TOKEN`
  / `notion.parentPageId` (or `DYDO_NOTION_PARENT_PAGE`). Reduce the manual friction we hit here.

## Framework hardening

- **`dydo init` scaffolds the FULL 2.0 setup + a parity test.** Agents/skills are already
  generated by `dydo sync` from role JSON + `Templates/mode-*.template.md`, and `sync-model.json`
  auto-seeds from its template — but the `.claude/workflows/*.js` (run-sprint, inquisition, the
  coming qa-loop) are **hand-authored and scaffolded by nothing**, so a fresh project lacks them
  and ours can drift silently. Make init ship them as embedded defaults, then add a parity test
  that runs init/sync into a temp and asserts the committed setup matches (no drift).
- **`qa-loop` workflow (self-healing QA gate).** Encapsulates *inquisition -> fixes (sprint
  code/review loop) -> re-inquisition -> ... -> clean*. Keep the inquisitor maximally thorough
  (raise everything). Add a **judge** stage that triages CONFIRMED findings by importance
  (blocker / nit-backlog / needs-human) — the termination arbiter, distinct from the existing
  truth-check "verify". HARD RULE: any judgment call is **raised to the human, never assumed**.
  Terminates on judge-clean, human-needed, or a round cap.
- **Fix the Workflow-tool permission bug.** A control character in the saved `.claude/workflows/*.js`
  now trips the approval-dialog validator (post-restart permission-mode change), blocking
  `/run-sprint` and `/inquisition` entirely. Clean the offending char(s) from the scripts.
