# 023 — Backlog and Future-Features Doc Categories

> **Superseded by [DR 034 — PM Record Taxonomy](./034-pm-record-taxonomy.md) (2026-07-06).**
> `backlog/` and `future-features/` are no longer convention-only `type: context` doc categories.
> `backlog` is now a **`Task` status** (partition `tasks/backlog/`); `future-features/` is its own
> **`FutureFeature`** spine DB. The backlog-vs-task *separate-record* split in §5 collapses into one
> `Task` lifecycle. Read this record for history; DR 034 is authoritative.

`dydo/project/` gains two new doc-categories — `backlog/` and `future-features/` — added as defaults to every `dydo init`. `backlog/` holds identified, scoped, ready-ish work that isn't in flight and isn't yet an active task. `future-features/` holds far-out ideas without a concrete next step. Distinct from `issues/` (broken things). Convention-only: no CLI surface, no new frontmatter types. Items in both folders use the existing `type: context` value.

## Context

Today `dydo/project/` has one work-tracking doc-category (`issues/`) plus the transient `tasks/`. Two classes of work need durable homes neither covers:

- **Identified, scoped work not yet in flight** — concrete next-step slices that surface during implementation, thinking sessions, or inquisition reports. Not bugs (nothing broken). Not far-out (next-step-sized). Today they accumulate in scattered notes, reviewer comments, and orchestrator workspaces — easy to lose, hard to surface.
- **Far-out ideas without a concrete next step** — features worth keeping in mind so adjacent code is shaped to accommodate them later, but not in scope for the current version.

Both categories were validated in a downstream fork and proved their keep. Porting them as dydo defaults so every new project scaffolds them out of the box closes the gap framework-wide.

## Decision

### 1. Locations and names

Two new folders alongside the existing `issues/`:

- `dydo/project/backlog/` — identified, scoped, ready-ish work.
- `dydo/project/future-features/` — far-out ideas without a concrete next step.

Industry-standard Agile vocabulary for the first; descriptive for the second. Zero learning curve.

### 2. File format

Single slug-named `.md` file per item under either folder. Numbering stays reserved for `issues/`.

**`backlog/` required frontmatter:**

```yaml
---
area: <one of the existing area enum values>
type: context                              # existing enum value
status: open | in-flight | done | cancelled
created: <YYYY-MM-DD>
created-by: <agent-name-or-balazs>
origin: <one-line — where the item came from>
---
```

**`future-features/` required frontmatter:**

```yaml
---
area: <one of the existing area enum values>
type: context
---
```

Reusing `type: context` (rather than minting `type: backlog` / `type: future-feature`) keeps the convention purely documentary — no enum changes. The folder location is the discriminator.

**Optional `backlog/` frontmatter:**

```yaml
related-issues: [<issue-ids>]
related-decisions: [<decision-numbers>]
related-tasks: [<task-names>]              # populated on pickup
```

**Explicitly dropped:** `priority`, `estimated-scope`. Both drift and become rubber-stamping. What's next is decided at pickup, not stamped on the item.

### 3. Lifecycle

`future-features/` items are static — created when an idea surfaces, deleted when the idea is implemented or rejected. No status, no archive subfolder. The folder is a parking lot.

`backlog/` items move:

| State | Frontmatter | File location |
|---|---|---|
| Created | `status: open` | `dydo/project/backlog/<slug>.md` |
| Picked up | `status: in-flight`, populate `related-tasks: [<slug>]` | (same) |
| Completed | `status: done` | move to `dydo/project/backlog/done/<slug>.md` |
| Cancelled | `status: cancelled` | move to `dydo/project/backlog/done/<slug>.md` |

`done/` is the no-longer-live archive; the `status` field disambiguates completed from cancelled. The `done/` subfolder is created the first time an item is archived — not before. Mirrors `issues/resolved/` exactly — no new mental model.

`_index.md` for both folders is auto-generated by `dydo fix` (hub type), listing only live (non-`done/`) items.

### 4. CLI vs convention

**Convention-only.** No `dydo backlog create/list/done`, no `dydo future-features` command. Agents write files directly with the schema above; `dydo check` validates the `area`/`type` essentials (`type: context` is already accepted).

Adding CLI later is cheap; pulling it back if adopted prematurely is hard. Revisit if convention-only friction proves real.

### 5. Relationship to `dydo/project/tasks/`

A **backlog item** *describes* a unit of work — durable, captures intent + motivation + context.

A **task** (under `dydo/project/tasks/`) is an *active assignment* of work to an agent — transient by design, "not indexed; locate by name via `dydo whoami` or `dydo agent status`".

**Pickup transition:** an orchestrator or balazs picks a backlog item → creates a task → flips the backlog file's frontmatter (`status: in-flight`, `related-tasks: [<slug>]`).

**On completion:** backlog `status: done`, file moves to `backlog/done/`. The task file is the durable record of *what was done*; the backlog file is the durable record of *what was intended*. Both survive — they answer different questions.

`future-features/` does not interact with tasks. When an item is ready to schedule, it either graduates into `backlog/` or directly into a task.

### 6. Category boundary

The convention answers "where does this go?" by:

> If the thing is broken → `issues/`. If the thing is identified work not yet in flight → `backlog/`. If the thing is a far-out idea without a concrete next step → `future-features/`.

Borderline items stay where they were filed unless they obviously cross the boundary. Don't migrate well-described entries just because a new category is available — that erodes the boundaries.

### 7. Role permissions

Roles that gain `dydo/project/backlog/**` write access (via `dydo/_system/roles/*.role.json`):

- **orchestrator** — picks up backlog items, transitions, archives.
- **co-thinker** — captures concrete next-step slices that surface during thinking sessions.
- **code-writer** — files non-blocking follow-ups observed during implementation.
- **judge** — files items from inquisition reports (parallel to how judge writes `project/issues/`).

Roles that remain read-only on `backlog/` (their existing escalation pattern handles it):

- **reviewer** — surfaces follow-ups in review notes; the dispatcher files.
- **inquisitor** — surfaces follow-ups in the inquisition report; the judge files.

`future-features/` gets no role-specific grants. Items there are rarely filed mid-task; docs-writer's existing `dydo/project/**` grant covers occasional updates. Agents who want to file a future-features item from another role escalate to the user.

### 8. Mode-file updates

One tight line per file, appended next to each role's existing `issues/` guidance:

- **code-writer:** "Non-blocking follow-ups (not bugs) skip approval — file directly to `dydo/project/backlog/<slug>.md` (`type: context`)."
- **co-thinker:** "Concrete next-step slices → `dydo/project/backlog/<slug>.md`; far-out ideas → `dydo/project/future-features/<slug>.md`."
- **orchestrator:** "Non-blocking sub-agent findings → `dydo/project/backlog/` (not `issues/`); on pickup flip `status: in-flight`, on done move to `backlog/done/`."
- **judge:** "Confirmed-but-non-blocking findings → `dydo/project/backlog/<slug>.md` instead of `issues/`."

Same-or-smaller weight than the existing `issues/` mentions. No sub-sections.

### 9. Worktree junctions

Both `dydo/project/backlog/` and `dydo/project/future-features/` join `dydo/project/issues/` and `dydo/project/inquisitions/` as shared subpaths in `WorktreeCommand.JunctionSubpaths`. Agents working in worktrees see backlog/future-features state from the main tree, and filing from a worktree lands in the main tree — no merge needed for these convention-only categories.

## Alternatives Considered

**Ship only `backlog/` and defer `future-features/`.** Rejected: the two categories answer adjacent questions and have validated together. Splitting the rollout would mean a second decision-doc, a second PR, and a window where the boundary rule references a folder that doesn't exist yet. Same convention, separate folder — cheap to ship together.

**`queued/` or `queued-slices/`.** More dydo-native sounding; explicit about "ready to do, not in flight." Rejected: `backlog/` carries the same meaning with zero learning curve and matches what mature Agile practice already calls this category.

**`follow-ups/`.** Descriptive of the common origin. Rejected: too narrow. Backlog includes net-new items too — small scoped ideas that aren't follow-ups to anything.

**Ship a CLI on day one.** Stand up `dydo backlog create/list/done` mirroring the issue CLI. Rejected: not load-bearing for adoption — agents can write files directly. Adding CLI is cheap later; pulling it back if adopted prematurely is hard.

**Convert-and-delete backlog items on pickup.** When picked up, run `dydo task create` and delete the backlog source file. Rejected: loses the intent + motivation context that lived in the backlog file. Tasks are transient by design; the durable record needs a home.

**Include `priority` and `estimated-scope` in required frontmatter.** Rejected: both drift, both become rubber-stamping. "What's next" is decided at pickup, not stamped on the item.

**Grant `dydo/project/future-features/**` to the same four roles as `backlog/`.** Rejected: future-features items are rarely filed mid-task — the bar for "this should be tracked for later, but not now" is set during planning sessions, not during code-writing. Docs-writer's existing `dydo/project/**` grant covers the occasional update; everything else escalates to the user.

## Consequences

- **Framework changes are additive only** — folder scaffolding, two folder-meta templates, four role-permission additions, two worktree junctions, four one-line mode-file edits.
- **`type: context` reused; no enum changes.** The folder location is the discriminator.
- **Land empty.** No seed items added to `backlog/`. Pre-existing entries in `future-features/` continue to use their original frontmatter (`type: concept`); the convention applies to new entries.
- **`done/` subfolder is created lazily** when the first item is archived — keeps the initial scaffold clean.
- **Existing dydo projects don't gain the folders automatically.** New projects scaffold both via `dydo init`. Existing projects can copy `_backlog.template.md` and `_future-features.template.md` from `_system/templates/` and create the folders manually, or run a workspace migration (out of scope for this decision).
