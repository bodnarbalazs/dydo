### Findings

#### 1. Watchdog kills fresh re-dispatches in the gap between polls (root cause)
- **Category:** bug
- **Severity:** critical
- **Type:** tested-by-evidence (audit-log forensics + code analysis from co-thinker brief)
- **Evidence:**
  - `Services/WatchdogService.cs:248` — `PollAndCleanup` kills any non-shell process whose command line contains the literal `"{agentName} --inbox"` whenever it observes `auto-close: true` AND `status: free` in `state.md`. The read at `:412 ParseStateForWatchdog` uses `File.ReadAllText` with no lock against `AgentRegistry.WriteStateFile`. Polling cadence is 10s.
  - `Services/AgentRegistry.cs:492-502 ReleaseAgent` — flips `status: free` but **does not** clear `AutoClose`. By design (the field signals the watchdog to kill claude after release).
  - `Services/AgentRegistry.cs:1334-1341 SetDispatchMetadata` — re-asserts `auto-close: true` on a new dispatch.
  - **Race timeline (release-then-redispatch):**
    1. Agent X releases. State = `free + auto-close: true`.
    2. Orchestrator immediately re-dispatches X. `ReserveAgent` sets `status: dispatched` under lock; new terminal launches a new claude.
    3. Watchdog poll catches the brief `free` window from step 1 (or polls during the lock-released gap before step 2's write completes), then proceeds with kill loop.
    4. Kill loop scans processes by command-line pattern `{agentName} --inbox`. The **new** claude.exe matches because its prompt is identical (`TerminalLauncher.GetClaudePrompt` always returns `"{agent} --inbox"`).
    5. New claude.exe killed. PowerShell wrapper (Windows) has no `-NoExit` fallback, so the terminal closes silently.
    6. Agent state stays `working` (set by the new claim during step 2). No `Release` event is logged. Exact match for the user's symptom.
  - **Removed defense (introducing commit `b60b258`, 2026-03-27, "Fix auto-close race condition in watchdog PollAndCleanup"):** Removed the `PendingCleanup` two-poll deferral. The deferral required an agent be observed `free + auto-close + processes-running` on **two consecutive polls** before killing — that 10s window was sufficient to absorb a re-dispatch. Commit message claims the original phantom-close motivation "was fixed separately" (`63b71bc`), but `63b71bc` only addresses the case where shells are still cleaning up — it does *not* protect a re-dispatched fresh claude.
  - **Amplifying commit (`d2a3ba2`, 2026-04-24, "bugfixes (auto-close regression & worktree race)"):** Re-anchored the watchdog from `GetParentPid(Environment.ProcessId)` (the dispatcher, which dies in seconds) to `FindAncestorProcess("claude")` (the orchestrator's claude session). Diff at `Services/WatchdogService.cs:114-122,200-206`. The watchdog now lives for the entire orchestrator session, so the kill race fires on every re-dispatch instead of only when a watchdog happens to overlap with one.
  - **Empirical death rate, step change after `d2a3ba2`** (sessions claimed but never released, in `dydo/_system/audit/2026/`):
    - Pre-`d2a3ba2` (2026-04-14 → 2026-04-23): **11 deaths over 10 days** ≈ 1.1 / day.
    - Post-`d2a3ba2` (2026-04-26 → 2026-04-28): **11 deaths over 3 days** ≈ 3.7 / day.
    - LC project mirrors the pattern: **4 death-shaped sessions on 2026-04-28 alone**.
    - Of the 25 DynaDocs sessions classified as "death-shaped" (claimed without release), the terminating events are scattered across `Read`, `Write`, `Edit`, `Bash`, `Role`, `Blocked` — i.e. mid-work, no shutdown handshake, consistent with external `Process.Kill()`.
  - **Window B — `ClearAutoClose` clobbers concurrent writes (compounds Window A; verified by Brian's scout `agent-deaths-watchdog-quality`).** `Services/WatchdogService.cs:285-295` does `ReadAllText → string.Replace → WriteAllText` with no lock. If `WriteStateFile` from `ReserveAgent`/`SetDispatchMetadata`/`SetupAgentWorkspace` lands between the read and the write, the watchdog's snapshot (still showing `status: free`) is rewritten back over the dispatcher's `Dispatched`/`Working` content. State is silently rolled back to `free + auto-close: false`. Subsequent ticks won't kill (auto-close is now false) but `dispatched-by`, `since`, and the role/task fields are corrupt. Worst-case: the next `ReserveAgent` reads the rolled-back `Free` state and launches **another** tab → double-launch and orphaned terminals.
  - **Test code:** the same race is reproducible by hand using the recipe at the bottom of this finding (do NOT run with live in-flight agents):
    ```bash
    # Pre-conditions: orchestrator session active, agent X free, watchdog running.
    dydo dispatch --no-wait --auto-close --role planner --task race-repro --brief "trivial planning"
    # X claims, writes a tiny plan, releases.
    # Within ~5 seconds:
    dydo dispatch --no-wait --auto-close --role code-writer --task race-repro --brief "implement"
    # New X terminal vanishes within 10–20s when the watchdog fires.
    ```
- **Judge ruling:** CONFIRMED
- **Files examined:** Services/WatchdogService.cs (lines 9-12, 114-122, 200-215, 238-295, 412-433), Services/AgentRegistry.cs (lines 130-156, 311-317, 335, 353-363, 480-510, 1334-1341, 1350-1404, 1971-2002), commits b60b258, 63b71bc, b2db397, d2a3ba2 (read full diffs), audit logs in `dydo/_system/audit/2026/*.events`.
- **Independent verification:** Walked the kill-loop control flow: `ParseStateForWatchdog` (line 248) reads state without acquiring `TryAcquireLock` (line 1978), the kill decision at line 249 acts on that snapshot with no recheck, and `Process.Kill()` runs at line 271. Confirmed `ClearAutoClose` (lines 285-295) is unlocked RMW: `ReadAllText → string.Replace → WriteAllText`. Read the b60b258 diff: it removed the `PendingCleanup` set (the two-poll deferral) without compensating protection. Read the 63b71bc diff: it only addresses the "shell processes still cleaning up" timing — does NOT cover redispatched live claude. Read the b2db397 diff: it added `ShellProcessNames` skip — also does NOT cover live redispatched claude (claude.exe / node is non-shell). Read the d2a3ba2 diff: confirmed re-anchor from `GetParentPid(Environment.ProcessId)` (the dying dispatcher) to `FindAncestorProcess("claude")` (long-lived orchestrator session). Counted death-shaped sessions (claim without release): pre-d2a3ba2 (2026-04-14 → 2026-04-23) = 11 deaths over 10 days = 1.1/day; post-d2a3ba2 (2026-04-26 → 2026-04-28) = 11 deaths over 3 days = 3.7/day. Subtracting the two in-progress 2026-04-28 sessions (one is this judge session) the post-rate is still ~3.0/day, ~3× the pre-rate. Step change confirmed.
- **Alternative explanations considered:** (a) Could the kill loop re-read state before `Process.Kill()` and abort on stale decision? No — there is no recheck; the snapshot is bound at line 248. (b) Could `string.Replace` on the watchdog side be benign if no rollback occurs because it only flips one field? No — when the dispatcher's write lands between the watchdog's `ReadAllText` and `WriteAllText`, the watchdog rewrites the entire file with the stale snapshot's content (only `auto-close: true→false` swapped). The rest of the dispatcher's fields (`status`, `role`, `task`, `dispatched-by`) are silently rolled back. (c) Could the `ShellProcessNames` skip protect the new claude? No — claude on Windows is `node` and on Linux is `claude` (per inquisitor + ProcessName). Neither is in the skip set, by design (the skip is intended to spare bash wrappers, not claude). (d) Could the death-rate step change have a confounder? Possibly — the user reported similar issues across two projects which lines up with the mechanism; the 3× step change is on the same audit pool with no other watchdog change in that interval; no other obvious confounder.
- **Issue:** #0121

#### 2. Watchdog kills the Linux terminal emulator itself (independent root cause; user's `Desktop/LC` + cross-platform reports)
- **Category:** bug
- **Severity:** critical
- **Type:** tested-by-evidence (audit-log forensics of LC project + code analysis from scout `agent-deaths-watchdog-quality`)
- **Evidence:**
  - `Services/TerminalLauncher.cs:73-90` — every `LinuxTerminals` entry launches `gnome-terminal -- bash -c "...claude '{agentName} --inbox'..."` (and `konsole`, `xfce4-terminal`, `alacritty`, `kitty`, `wezterm`, `tilix`, `foot`, `xterm` analogously). The terminal-emulator process inherits the `bash -c "..."` argv, including the `claude '{agentName} --inbox'` substring.
  - `Services/ProcessUtils.CommandLine.cs:110` — Linux watchdog reads `/proc/{pid}/cmdline`, which preserves argv. `FindProcessesByCommandLine` at `WatchdogService.cs:254` returns the terminal-emulator PID alongside (or instead of) the claude PID.
  - `Services/WatchdogService.cs:9-12` — `ShellProcessNames = { "powershell", "pwsh", "bash", "sh", "cmd", "zsh" }`. The Linux terminal-emulator process names — `gnome-terminal`, `konsole`, `xfce4-terminal`, `alacritty`, `kitty`, `wezterm`, `tilix`, `foot`, `xterm` — are **not in this set**.
  - `Services/WatchdogService.cs:266-272` — the skip filter (`ShellProcessNames.Contains(procName)`) therefore does not protect terminal emulators. `Process.Kill()` runs against the emulator PID. Killing the emulator takes down the bash child and the claude grandchild together — i.e. **the entire tab vanishes**, no graceful close, no release event.
  - This explains the user's `C:/Users/User/Desktop/LC` deaths cleanly (LC is a Linux project per `dydo.json`/`Dockerfile` evidence) and any Linux DynaDocs sessions. It is **independent of finding #1**: even without the redispatch race, every legitimate auto-close kill on Linux destroys the terminal that is hosting the tab. Combined with finding #1, the same kill happens *before* the agent has even finished, mid-work.
  - **Mac is robust by accident.** `MacTerminalLauncher` uses `osascript -e 'tell app "Terminal" to do script ...'`. `osascript` carries the prompt in argv but exits within milliseconds. `Terminal.app` / `iTerm` carry no script content in argv (script delivered via Apple Events). On a slow watchdog tick this is safe; under shorter polling intervals the osascript window could still race. `osascript` should be added to `ShellProcessNames`.
  - **Windows is robust.** `pwsh`/`powershell` are in `ShellProcessNames` and are skipped. `wt.exe` (process name `wt`) is short-lived. Claude on Windows is `node.exe` (process name `node`) — not in the skip list, which is the *intended* kill target.
- **Judge ruling:** CONFIRMED
- **Files examined:** Services/TerminalLauncher.cs (lines 73-90 = LinuxTerminals table), Services/WatchdogService.cs (lines 9-12 = ShellProcessNames, lines 251-274 = kill loop with skip filter), Services/ProcessUtils.CommandLine.cs (lines 100-119 = `FindProcessesByCommandLineLinux` reading `/proc/{pid}/cmdline`), Services/ProcessUtils.cs (lines 44-58 = `GetProcessName` returns `Process.ProcessName`).
- **Independent verification:** Read every entry of the `LinuxTerminals` table — all 9 entries (`gnome-terminal`, `konsole`, `xfce4-terminal`, `alacritty`, `kitty`, `wezterm`, `tilix`, `foot`, `xterm`) wrap the launch as `bash -c "...claude '{agent} --inbox'..."`. Confirmed by inspection that the `claude '{agent} --inbox'` substring is present in every variant. Verified `FindProcessesByCommandLineLinux` reads `/proc/{pid}/cmdline` (NUL-replaced with spaces) and substring-matches case-insensitively — Linux preserves argv in `/proc/{pid}/cmdline` for the lifetime of any process. Confirmed `ShellProcessNames` lacks all 9 emulators plus `osascript` (Mac) and `wt` (Windows defense-in-depth). Confirmed Mac is robust by accident: `MacTerminalLauncher` uses `osascript -e ...` which exits in milliseconds, so the watchdog rarely sees its argv (and `Terminal.app`/iTerm receive scripts via Apple Events, not argv). Windows is robust because `pwsh`/`powershell` are in the skip set and `wt` is short-lived.
- **Alternative explanations considered:** (a) Daemonizing terminals (gnome-terminal, konsole, xfce4-terminal, tilix may exit quickly after forwarding to a server) — even so, the standalone-process terminals (alacritty, kitty, foot, wezterm, xterm) are long-running ELF processes whose `/proc/{pid}/cmdline` carries the bash command for their entire lifetime. One affected terminal is sufficient to confirm the finding; multiple are affected. (b) Could the bash child be the only match (and bash IS skipped)? No — `FindProcessesByCommandLineLinux` returns ALL matching PIDs; the substring is present in BOTH the bash process's argv AND the terminal-emulator's argv (because the terminal was launched with the bash command in *its* argv). The skip filter exempts bash but not the emulator. (c) The user's deaths on the LC project (Linux per `dydo.json`/`Dockerfile`) match the cross-platform symptom and are independent of #1 — every legitimate auto-close kill on Linux destroys the host terminal, not just races.
- **Issue:** #0122

#### 3. `ReleaseAgent` leaves `auto-close: true` on disk — kill condition is set under no further re-dispatch protection
- **Category:** bug (root-cause partner of #1)
- **Severity:** high
- **Type:** obvious
- **Evidence:** `Services/AgentRegistry.cs:492-502 ReleaseAgent` clears `Status`, `Role`, `Task`, `Since`, `WritablePaths`, `ReadOnlyPaths`, `UnreadMustReads`, `UnreadMessages` — but not `AutoClose`. `CleanupAfterRelease` at `:574-589` doesn't either. The `ClearAutoClose` path lives in the watchdog (`WatchdogService.cs:285`) and runs only after the kill loop. So between release and the watchdog's next poll (up to 10s), the agent is in the lethal `free + auto-close: true` state. Clearing `AutoClose` on release would close the race even without re-introducing the deferral, because the second dispatch's `SetDispatchMetadata` is the only thing that should ever re-set `auto-close: true`.
- **Judge ruling:** CONFIRMED
- **Files examined:** Services/AgentRegistry.cs (lines 480-510 = `ReleaseAgent`, lines 574-599 = `CleanupAfterRelease`, lines 1334-1341 = `SetDispatchMetadata`, lines 1343-1404 = `UpdateAgentState` and `WriteStateFile`), Services/WatchdogService.cs (lines 285-295 = `ClearAutoClose`).
- **Independent verification:** Read the `ReleaseAgent` lambda character-by-character: `s.Status`, `s.Role`, `s.Task`, `s.Since`, `s.WritablePaths`, `s.ReadOnlyPaths`, `s.UnreadMustReads`, `s.UnreadMessages` are all reset; `s.AutoClose`, `s.WindowId`, `s.AssignedHuman`, `s.DispatchedBy`, `s.DispatchedByRole` are not touched. Read `CleanupAfterRelease` (cleans wait/reply/dispatch markers, modes dir, nudge files) — confirmed it doesn't touch state.md. Read `SetDispatchMetadata` — confirmed `auto-close: true` is set there only on dispatch. Confirmed via `WriteStateFile` template (line 1369 = `auto-close: {state.AutoClose...}`) that the field is written every time. So once `auto-close: true` is set on dispatch, only the watchdog's `ClearAutoClose` clears it.
- **Alternative explanations considered:** (a) Could leaving `auto-close: true` post-release be intentional, since the watchdog needs that signal to kill claude? Yes — that is the design. The bug is that the design assumes the watchdog will *always* observe `free + auto-close: true` before any redispatch and act first. That assumption fails under the redispatch race in finding #1. Clearing `AutoClose` on release inverts the contract: the dispatcher re-sets it on the next dispatch (already does, line 1334-1341), the watchdog clears it on observation (already does, line 285-295). The only state where `auto-close: true` would survive across release-without-kill is if a release happens between the watchdog's `ClearAutoClose` and the next dispatch — and that path doesn't exist by construction (release is the *first* event of the lifecycle gap, ClearAutoClose runs against `free`). (b) Confirmed this is a true partner bug to #1, not a duplicate — fixing #3 alone closes the redispatch race window for finding #1 even without acquiring locks. They are independent fixes with overlapping effect.
- **Issue:** #0123

#### 4. Windows launcher has no `-NoExit` fallback when claude exits unexpectedly
- **Category:** bug (independent failure mode + diagnosability)
- **Severity:** high
- **Type:** obvious
- **Evidence:** `Services/WindowsTerminalLauncher.cs:11-19` (per co-thinker brief; quoted line numbers confirmed against `Services/TerminalLauncher.cs:166-191`). The Windows command is `claude '{prompt}'; if ((dydo agent status) -match 'free') { exit 0 }`. There is no `else { exec pwsh }` branch and no `-NoExit` flag on `wt.exe`/`pwsh.exe`. Compare with the Linux equivalent at `Services/TerminalLauncher.cs:76-90` and `BashPostClaudeCheck` at `:134-135` which `exec bash` so the terminal stays open with diagnostic output if claude exits while the agent is not free. Consequence: any non-Release exit of claude.exe (watchdog kill, /exit, network drop, context limit, crash) on Windows produces a **silent terminal close** with no visible error and no stay-open shell — exactly matching "terminal closed mid-work without release". This explains why the user can't see what went wrong even when the underlying cause is finding #1.
- **Judge ruling:** CONFIRMED
- **Files examined:** Services/WindowsTerminalLauncher.cs (lines 11-68 = `GetArguments`), Services/TerminalLauncher.cs (lines 73-90 = LinuxTerminals table, lines 134-135 = `BashPostClaudeCheck`).
- **Independent verification:** Read `WindowsTerminalLauncher.GetArguments` from start to end. Confirmed:
  - Line 18: `var noExitFlag = autoClose ? "" : "-NoExit ";` — empty string when autoClose is set.
  - Lines 15-17: `var postClaudeCheck = autoClose ? "; if ((dydo agent status {agentName} 2>&1) -match 'free') { exit 0 }" : "";` — no else branch.
  - Line 67 (and the worktree variants at 38, 48, 63): the final assembled command is `{noExitFlag}-Command "...claude '{prompt}'{postClaudeCheck}"`. PowerShell's `-Command` mode exits when the script body completes; without `-NoExit` and without an else branch, any non-free exit of claude → script body falls through → PowerShell exits → terminal closes.
  - Compared to Linux `BashPostClaudeCheck` at TerminalLauncher.cs:134-135: `if dydo agent status {agent} | grep -q 'free'; then exit 0; fi; exec bash`. The trailing `; exec bash` keeps the terminal open with a diagnostic shell on non-free exit.
- **Alternative explanations considered:** (a) Could PowerShell's `-Command` mode be expected to keep the window open? No — `-Command` exits on script end by default; you need `-NoExit` to keep the window. (b) Could the user be expected to inspect logs elsewhere (e.g. claude's own log)? Possibly, but the inconsistency with Linux is the bug; diagnosis on Windows requires post-hoc log spelunking that Linux doesn't. (c) Independent of findings #1 / #2 — even if those root causes were fully fixed, claude itself can exit non-released for other reasons (`/exit`, network drop, context limit, crash) and Windows users would still see a silent close. Confirmed.
- **Issue:** #0124

#### 5. State-file read/write is unsynchronized between watchdog and registry
- **Category:** race condition / antipattern
- **Severity:** medium
- **Type:** obvious
- **Evidence:** `Services/AgentRegistry.cs:1350-1403 WriteStateFile` rebuilds the entire state file content and writes it via `File.WriteAllText` (no atomic-replace, no lock). `Services/WatchdogService.cs:412 ParseStateForWatchdog` reads via `File.ReadAllText`. The two paths share no lock. On NTFS a torn read returns a `Read` exception which `ParseStateForWatchdog` catches and returns `(false, false, null, null)` — that's a partial mitigation, but it does not protect against logically stale-but-syntactically-valid reads (e.g. reading the `free` content that was committed to disk a moment ago, then the kill loop running 200ms later by which point the registry has already written `dispatched`). Correct fix: take the same per-agent lock that `ReserveAgent`/`ReleaseAgent` use (`TryAcquireLock`/`ReleaseLock`) when reading state from the watchdog. Cited as recommendation #1 in co-thinker brief.
- **Judge ruling:** CONFIRMED
- **Files examined:** Services/AgentRegistry.cs (lines 1350-1404 = `WriteStateFile`, lines 1971-2002 = `TryAcquireLock` / lock implementation, lines 130-156 = `ReserveAgent` lock-bracketed), Services/WatchdogService.cs (lines 285-295 = `ClearAutoClose` unlocked RMW, lines 412-433 = `ParseStateForWatchdog` unlocked read).
- **Independent verification:** Confirmed `WriteStateFile` ends with a single `File.WriteAllText(statePath, content)` (line 1403) — no atomic temp-and-rename. Confirmed `TryAcquireLock` (line 1978) is a file-based exclusive lock at `.claim.lock` per agent workspace. Confirmed all three writers — `ReserveAgent` (line 124-156), `ClaimAgent` (line 231-256), `ReleaseAgent` (line ~470-510) — bracket their `UpdateAgentState` calls with `TryAcquireLock`/`ReleaseLock`. Confirmed the watchdog's three state touchpoints — `ParseStateForWatchdog` (line 412), `ClearAutoClose` (line 285), and the kill decision at line 248 — never call `TryAcquireLock`. The torn-read protection in `ParseStateForWatchdog` is a `try/catch` returning `(false, false, null, null)` which avoids spurious kills *only* on torn reads, not on stale-but-valid reads.
- **Alternative explanations considered:** (a) Could the watchdog hold the lock cause deadlock with the registry? No — `TryAcquireLock` is per-agent; the watchdog can take it for the duration of a single agent's poll iteration without holding it across agents. (b) Could it be intentional to skip the lock on the read side for performance? No — the watchdog ticks once every 10s, so lock contention is negligible. (c) Could atomic-replace alone (without the lock) close finding #1's Window A? No — atomic replace prevents torn reads but not the stale-decision-then-kill race; the watchdog can still read a `free` snapshot that was atomically committed a moment before the dispatcher's atomic `dispatched` write. The lock is the correct fix for both #1's Window A and the corrupting #1 Window B; atomic replace complements but does not substitute.
- **Issue:** #0125

#### 6. Co-thinker brief and watchdog-quality scout identify several adjacent issues
- **Category:** mixed
- **Severity:** low–medium (per finding)
- **Type:** obvious
- **Evidence:** See `dydo/agents/Brian/archive/20260428-144500/brief-agent-deaths.md` (co-thinker root-cause brief, preserved by archival on reclaim) and `dydo/agents/Brian/findings-watchdog-quality.md` (scout's full Q1–Q7 verdicts). Additional points the judge should rule on independently:
  - **Phantom-close defense regression — `b60b258`'s "fixed separately" claim is misleading.** The "separately fixed" change is `b2db397` (2026-03-13), which added the `ShellProcessNames` skip set. That handles only the bash-wrapper subset of phantom close — it does NOT close the redispatch race that the deferral was also protecting against. `b60b258` traded the dead-process kill race for a *live*-process kill race; the new race is strictly worse. The commit message should not have been treated as authoritative.
  - **Anchor PID null path → orphaned watchdog runs forever.** `FindAncestorProcess("claude")` (`Services/ProcessUtils.Ancestry.cs:37-56`) returns null when no Claude is in the ancestry (CLI tests, manual `dydo dispatch`, scheduled jobs). When null, `DYDO_WATCHDOG_ANCHOR_PID` is unset; `WatchdogService.cs:204-206` resolves `anchorPid = null`; the liveness check at `:215` becomes `false && …` = `false` and is *never* taken. The only remaining exits are `ProcessExit`/`CancelKeyPress`, which do not fire reliably on detached watchdogs. **The watchdog leaks until manual `dydo watchdog stop` or reboot.**
  - **Anchor pinned to first dispatcher's claude only.** `EnsureRunning` short-circuits when the PID file is live (`WatchdogService.cs:72-79`); only the FIRST dispatcher's resolved anchor matters. If that claude exits while later-dispatched agents are still mid-work, the watchdog dies and their auto-close cleanups never run. The next dispatch then spawns a new watchdog — non-deterministic lifecycle tied to whichever agent dispatched first.
  - **`FindAncestorProcess` substring match is too broad.** `ProcessUtils.Ancestry.cs:49` does `name.Contains("claude", OrdinalIgnoreCase)`. On Windows a process named `claudia.exe`, `claude-dev.exe`, or any third-party tool whose name contains "claude" will be picked as the anchor. Should be exact-match or `StartsWith("claude", …)`. Same broad-match concern at the env-var validation: a `DYDO_WATCHDOG_ANCHOR_PID` of 1 (init/System) would never appear "gone."
  - **Stale-working reclaim silently archives in-flight work.** `Services/AgentRegistry.cs:311-317` (path added by `215e8d6`). After a death from finding #1/#2 the agent appears `Working` past 5 min with dead session PID, so the next claim archives the whole workspace via `SetupAgentWorkspace`'s `ArchiveWorkspace`. Forensic recovery is possible (`dydo/agents/<agent>/archive/<ts>/`) — that's how the co-thinker's brief was recovered above — but agents lose context.
  - **Watchdog has zero structured logging.** `PollAndCleanup` swallows kill-loop errors (`WatchdogService.cs:273`); the loop also swallows top-level exceptions (`:223-226`); there is no log of which PIDs were killed, which state was observed, or whether the anchor was ever resolved. The watchdog runs as a background process whose only visible artifact is the side effect (terminal closes). Until logging is added, this class of bug is undiagnosable in production. Bare minimum: write to `dydo/_system/.local/watchdog.log` on every kill with PID, command line, and the state snapshot used.
  - **`ShellProcessNames` is missing common shells.** Per scout: `fish`, `dash`, `tcsh`, `csh`, `nu`, `ksh` are not in the set. Linux users running any of these as their dispatcher shell would have an additional skip-bypass surface. Low severity given the dominant shells are covered, but cheap to extend.
  - **Substring CLI matching has an undocumented load-bearing invariant.** `WatchdogService.cs:251` builds `$"{agentName} --inbox"`. The trailing space + `--inbox` token guards against prefix collisions like `Jack` vs `Jacky` (per scout Q1). If anyone changes the dispatch prompt format (`--no-inbox`, longer suffix), the implicit boundary disappears. There is no comment at the construction site warning that the trailing token is collision-safety. Low severity but should be either documented or replaced with a token-aware match.
- **Judge ruling:** CONFIRMED (per-subitem; this is a bundle, ruled on independently below)

- **Per-subitem rulings:**

### Hypotheses Not Reproduced
- **Multiple watchdogs racing.** `EnsureRunning` is gated by `FileMode.CreateNew` on a PID file (`WatchdogService.cs:89`). Scout walked through the two-dispatcher and stale-PID-file races and verified safety on both NTFS and ext4.
- **Queue advancement causing kills.** `PollQueues` (`WatchdogService.cs:301`) does not kill processes; it only relaunches deferred queue items when the active entry's PID dies. Skips agents in `Working/Reviewing/Dispatched/Queued`.
- **Cross-agent CLI substring collision.** `ProcessUtils.FindProcessesByCommandLine` is substring-matched, but the trailing `" --inbox"` token in `WatchdogService.cs:251` guards against prefix collisions (`Jack` vs `Jacky`). Currently inert; the implicit invariant is now logged in finding #6 as a footgun if the dispatch prompt format ever changes.

### Suggested fixes (ordered by leverage; not implemented — inquisition only)
1. **Take the per-agent registry lock for the entire watchdog poll on each agent.** `PollAndCleanup` should acquire `AgentRegistry.TryAcquireLock(agentName)` before reading state, hold it across the kill loop, and release after `ClearAutoClose`. Closes finding #1 (Window A: stale-decision kill) AND finding #1 Window B (`ClearAutoClose` clobber) AND most of finding #5. Test by holding the lock from a registry write while triggering a watchdog poll and asserting no kill occurs.
2. **Resolve the actual claude (Linux: node) target instead of killing every match.** In `PollAndCleanup`, restrict kills to processes whose `ProcessName` is `claude` on Mac/Linux or `node` on Windows (or, more durably, the process whose parent chain includes the matched bash). This closes finding #2 cleanly. Alternative: extend `ShellProcessNames` with `gnome-terminal`, `konsole`, `xfce4-terminal`, `alacritty`, `kitty`, `wezterm`, `tilix`, `foot`, `xterm` (and `osascript` for Mac, `wt` for Windows defense-in-depth). The "skip emulators" approach is one-line; the "kill the actual target only" approach is more durable.
3. **Clear `AutoClose` on `ReleaseAgent`.** Smallest single change: add `s.AutoClose = false` to the lambda at `AgentRegistry.cs:492-502`. The "kill old claude" path then runs at most once per dispatch (set on dispatch, cleared on release-or-watchdog-kill) and the race window in finding #1 vanishes even without (1). Test by asserting `auto-close: false` on disk after release.
4. **Switch `WriteStateFile` to atomic temp-file rename.** `File.WriteAllText(temp, content); File.Move(temp, dest, overwrite: true)`. Atomic on POSIX (`rename(2)`) and on NTFS via `MoveFileEx(MOVEFILE_REPLACE_EXISTING)`. Eliminates torn-read class of bugs; complements (1).
5. **Add `-NoExit`/stay-open fallback to Windows launcher.** Mirror Linux `exec bash`. Independent fix that addresses diagnosability of finding #4 even after #1–#3 land.
6. **Add minimal watchdog logging.** Append to `dydo/_system/.local/watchdog.log` with anchor PID, kill timestamps, matched PIDs, command lines, and the state snapshot used. Necessary for detecting future regressions of this class.
7. **Tighten anchor selection.** Replace `Contains("claude", OrdinalIgnoreCase)` with exact-match in `ProcessUtils.Ancestry.cs:49`; reject `DYDO_WATCHDOG_ANCHOR_PID <= 1`; on null anchor, log + fall back to a max-watchdog-age timeout exit (e.g. 24 h) so an orphaned watchdog cannot leak indefinitely.
