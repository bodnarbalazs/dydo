# Documentation Model

The JITI (Just-In-Time Information) philosophy and how dydo structures documentation for AI consumption. In dydo, documentation **is** the memory — agents re-read it each session to onboard themselves without human hand-holding.

---

## JITI: Just-In-Time Information

Agents don't read everything upfront. They follow a funnel, reading only what's relevant to the current task. This keeps context windows focused and avoids overloading the agent with unrelated detail.

The principle: navigate to what you need, when you need it. Leave out what's not relevant.

---

## The Documentation Funnel

Every session follows this reading path:

```
entry file → index.md → hub files → task-specific docs
```

1. **Entry file** (`CLAUDE.md` / `AGENTS.md`) — loaded automatically by the platform; points every session at the docs and the skills.
2. **`index.md`** — dydo orientation: the knowledge tree, the records, the skills model.
3. **Hub files** (`_index.md`) — each folder's table of contents; navigate down only where the task leads.
4. **Task-specific docs** — whatever the work needs: architecture docs, decision records, reference material.

The *how* of working arrives the same way: skills load their methodology when invoked, not upfront. Each layer narrows the focus — by the time work starts, the session has exactly the context it needs.

---

## Folder Structure

Documentation lives in four top-level folders, each serving a different purpose:

| Folder | Contains | Read When |
|--------|----------|-----------|
| `understand/` | Domain concepts, architecture, system overview | Starting a task, needing context |
| `guides/` | How-to instructions, coding standards | Doing specific work |
| `reference/` | CLI reference, API specs, configuration | Looking up exact details |
| `project/` | Decisions, changelog, tasks, pitfalls | Needing history or project context |

---

## File Conventions

### Hub files (`_index.md`)

Every folder has an `_index.md` — the table of contents for that area. These are auto-generated by `dydo fix` and link to every document in the folder. They should not be manually edited — changes will be overwritten.

### Folder meta files (`_foldername.md`)

Direct children of main folders have a meta file (e.g., `guides/api/_api.md`) describing the subfolder's purpose. The first sentence of the summary is extracted and displayed in hub file links.

### Frontmatter

Every document starts with YAML frontmatter:

```yaml
---
area: understand
type: concept
---
```

Required fields: `area` (which section) and `type` (what kind of document). Optional: `status` and `date` (for decisions and changelogs).

## The Doc Graph

Documents link to each other through relative markdown links, forming a navigable graph. Use `dydo graph` to explore connections:

```bash
dydo graph dydo/understand/architecture.md             # Outgoing links
dydo graph dydo/understand/architecture.md --incoming   # What links here
dydo graph dydo/understand/architecture.md --degree 2   # Two-hop connections
```

Use `dydo graph stats` to see which documents are most referenced — useful for identifying central docs.

---

## Validation

Dydo enforces structural consistency. Run before committing:

```bash
dydo check    # Find issues: broken links, missing frontmatter, orphans
dydo fix      # Auto-fix: kebab-case renaming, wikilink conversion, missing hub files
```

Validation checks: frontmatter fields, naming conventions, link integrity, hub file presence, folder meta files, and orphan detection (every doc must be linked from somewhere).

---

## Why Markdown

Markdown is the right format because it's human-readable, git-diffable, and AI-parseable. Agents can read it natively without tooling. Humans can browse it in Obsidian, VS Code, or GitHub. Changes show up cleanly in diffs and pull requests.
